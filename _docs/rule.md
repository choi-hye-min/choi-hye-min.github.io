# 기술블로그 포스트 작성 규칙

## 목적

AI 또는 자동화 시스템이 `_posts`에 기술블로그 글을 작성할 때, 기존 블로그의 형식과 톤을 유지하면서도 독자가 바로 이해하고 실무에 적용할 수 있는 글을 만들기 위한 규칙이다.

이 문서는 단순히 긴 글을 만드는 것이 아니라, `초보자도 이해할 수 있는 설명`, `실전 코드`, `주의사항`, `참고자료`가 균형 있게 들어간 글을 작성하는 것을 목표로 한다.

---

## 파일 작성 규칙

1. **파일 위치**
   - 일반 포스트는 `_posts/{YYYY}` 폴더에 작성한다.
   - 예: `_posts/2026/2026-06-20-langgraph-core-concepts.md`

2. **파일명**
   - 기본 형식: `YYYY-MM-DD-{slug}.md`
   - `YYYY-MM-DD`: 작성 날짜
   - `{slug}`: 글의 주제를 나타내는 소문자 영문, 숫자, 하이픈 조합
   - 좋은 예: `2026-06-20-langgraph-core-concepts.md`
   - 피할 예: `2026-06-20-post20260620-0.md`
   - 같은 날짜에 여러 글을 작성하더라도 각 글에 서로 다른 의미 있는 slug를 사용한다.
   - 기존 포스트는 이전 URL을 유지하기 위해 파일명을 변경하지 않는다.
   - 포스트 전용 이미지는 `images/posts/{slug}/`에 두는 것을 권장한다.

3. **Front matter**
   - 모든 새 포스트는 아래 형식을 기본으로 사용한다.

   ```yaml
   ---
   layout: post
   title: "포스트 제목"
   date: YYYY-MM-DD HH:MM:SS +0900
   categories: [카테고리1, 카테고리2]
   tags: [태그1, 태그2, 태그3]
   ---
   ```

   - `layout`은 `post`로 고정한다.
   - `title`은 본문 내용을 구체적으로 드러내야 한다.
   - `date`는 작성 시점의 한국 시간 `+0900`을 사용한다.
   - `categories`는 1~4개를 권장한다.
   - `tags`는 3~8개를 권장한다.
   - `tags`는 검색과 묶음 탐색에 쓰이므로 너무 긴 문장형 태그를 피한다.
   - 대표 이미지가 있을 때만 `img: path/to/image.png`를 추가한다.
   - 검색 결과나 공유 카드에 사용할 설명이 필요하면 `description`을 1~2문장으로 추가한다.
   - 기존 URL을 변경한 경우에만 `redirect_from`에 이전 URL을 기록한다.
   - 검색엔진에 노출하지 않을 페이지나 임시 글에만 `sitemap: false`를 사용한다.

4. **새 포스트 생성기 사용**
   - 새 포스트는 프로젝트 루트에서 아래 명령으로 생성한다.

   ```bash
   ruby scripts/new_post.rb "포스트 제목" english-slug
   ```

   - 생성기는 한국 시간 기준 연도 폴더, 작성 날짜, front matter를 자동으로 구성한다.
   - slug에는 소문자 영문, 숫자, 단일 하이픈만 사용한다.
   - 생성된 front matter의 빈 `categories`, `tags`를 본문 작성 전에 채운다.
   - 같은 경로의 파일이 이미 있으면 덮어쓰지 않으므로 다른 slug를 사용한다.

## 제목 규칙

1. 제목은 독자가 얻을 수 있는 내용을 분명히 보여준다.
   - 좋은 예: `Spring WebFlux 예외 처리 패턴과 Kotlin 예제`
   - 좋은 예: `Kafka Consumer 재시도와 DLQ 구성 방법`
   - 피할 예: `개발자가 꼭 알아야 할 모든 것`

2. 과장된 표현은 피한다.
   - `완전정복`, `완벽 가이드`, `모든 것`, `끝판왕` 같은 표현은 기본적으로 사용하지 않는다.
   - 꼭 넓은 범위를 다룬다면 `정리`, `가이드`, `실전 패턴`, `기초부터 실전까지` 정도로 낮춰 쓴다.

3. 제목은 너무 길게 만들지 않는다.
   - 권장 길이: 20~45자
   - 기술명, 문제 상황, 해결 방향이 들어가면 충분하다.

## 본문 기본 구조

새 포스트는 가능하면 아래 구조를 따른다. 주제에 맞지 않는 섹션은 생략할 수 있지만, `요약`, `본문`, `실전 예제`, `결론`, `참고자료`는 기본 포함을 권장한다.

```markdown
> 이 글에서 다루는 문제를 1문장으로 설명한다.
> 독자가 배울 핵심 내용을 1문장으로 설명한다.
> 실무 적용 관점에서 얻을 수 있는 결과를 1문장으로 설명한다.

## 주제의 배경과 문제 상황

## 핵심 개념

## 동작 방식 또는 구조

## 실전 예제

## 자주 하는 실수와 주의사항

## 실무 적용 팁

## 결론 및 도움말

## 참고자료/레퍼런스
```

## 본문 작성 가이드

1. **도입 요약**
   - 포스트 시작 부분에 `>`로 시작하는 2~3줄 요약을 작성한다.
   - 요약에는 `문제`, `다루는 내용`, `독자가 얻는 결과`가 들어가야 한다.

2. **독자 수준**
   - 기본 독자는 주니어 또는 해당 기술을 처음 실무에 적용하는 개발자로 본다.
   - 단어를 쉽게 풀어 쓰되, 부정확하게 단순화하지 않는다.
   - 처음 등장하는 약어는 한 번 풀어 쓴다. 예: `DLQ(Dead Letter Queue)`

3. **문단**
   - 한 문단은 3~5문장 이내로 유지한다.
   - 한 문장에 개념을 너무 많이 넣지 않는다.
   - 설명은 `정의 → 왜 필요한가 → 어떻게 쓰는가 → 주의할 점` 순서로 전개한다.

4. **헤딩**
   - 본문 주요 섹션은 `##`를 기본으로 사용한다.
   - 세부 항목은 `###`를 사용한다.
   - `####` 이하 헤딩은 세부 구조가 더 필요할 때만 제한적으로 사용한다.
   - 섹션 번호는 긴 글에서만 사용한다. 번호를 붙이면 끝까지 일관되게 유지한다.
   - 자동 목차는 `##`, `###`만 수집하므로 주요 구조에 `####`, `#####`를 사용하지 않는다.

5. **목차**
   - 포스트 목차는 `##`, `###` 헤딩을 기준으로 자동 생성된다.
   - 데스크톱에서는 우측 고정 목차, 모바일에서는 접이식 목차로 표시된다.
   - 목차에 나타나야 하는 섹션은 반드시 `##` 또는 `###`로 작성한다.

6. **코드 예제**
   - 서버/백엔드 예제는 가능하면 Kotlin을 사용한다.
   - Java, YAML, bash, SQL 등 다른 언어가 더 자연스러운 경우에는 해당 언어를 사용한다.
   - 모든 코드블록에는 언어를 명시한다.

   ```kotlin
   fun main() {
       println("example")
   }
   ```

   - 코드는 실행 흐름을 이해할 수 있을 만큼 완성된 형태로 작성한다.
   - 예제 코드 앞뒤에 `무엇을 보여주는 코드인지`, `실무에서는 어디를 바꿔야 하는지`를 설명한다.
   - 보안 토큰, 실제 비밀번호, 개인정보처럼 민감한 값은 예제로도 쓰지 않는다.
   - 일반 코드블록에는 실제 줄 수에 맞는 줄 번호가 자동으로 표시되므로 코드 안에 줄 번호를 직접 작성하지 않는다.
   - 긴 코드 행은 코드 영역 내부에서 가로 스크롤되므로 가독성을 해치는 임의 줄바꿈을 넣지 않는다.
   - 모든 여는 코드 fence에는 대응하는 닫는 <code>```</code>가 있어야 한다.
   - `mermaid` 코드블록은 다이어그램으로 렌더링되며 일반 코드 줄 번호 대상에서 제외된다.

7. **표와 목록**
   - 비교가 필요한 내용은 표를 사용한다.
   - 절차, 체크리스트, 장단점은 목록을 사용한다.
   - 목록만 길게 이어지는 글은 피하고, 목록 뒤에는 해석과 적용 기준을 덧붙인다.
   - 표의 셀 내용은 짧고 직접적으로 작성하고, 코드나 명령은 인라인 코드로 표시한다.
   - 표는 2~5개 열을 권장한다. 열이 많거나 설명이 길면 표를 나누거나 목록으로 전환한다.
   - 표는 본문 가로폭을 채우며 모바일에서는 필요할 때 표 내부에서 가로 스크롤된다.

8. **다이어그램**
   - 아키텍처, 메시지 흐름, 요청 처리 흐름은 Mermaid 다이어그램을 사용할 수 있다.
   - Mermaid는 아래처럼 언어가 `mermaid`인 fenced code block으로 작성한다.

   ````markdown
   ```mermaid
   flowchart LR
       Client --> API
       API --> Service
       Service --> Database
   ```
   ````

   - 노드 이름은 짧게 유지하고 복잡한 흐름은 여러 다이어그램으로 나눈다.
   - 다이어그램 뒤에는 반드시 글로 해석을 붙인다.
   - Jekyll 빌드만으로 Mermaid 문법 오류를 확인할 수 없으므로 로컬 브라우저에서 SVG로 렌더링되는지 확인한다.

9. **결론**
   - 마지막에는 `## 결론 및 도움말` 섹션을 둔다.
   - 결론은 `>` 인용문으로 1~2문단 작성한다.
   - 새 내용을 추가하기보다 핵심 판단 기준과 다음 학습 방향을 정리한다.

10. **참고자료**
    - 공식 문서를 1개 이상 포함하는 것을 권장한다.
    - 블로그/예제 저장소를 넣을 때는 공식 문서보다 보조 자료로 둔다.
    - 존재 여부가 불확실한 링크나 출처가 모호한 인용은 쓰지 않는다.

## 중복 방지 규칙

1. **파일과 front matter**
   - 기존 글과 같은 파일명을 사용하지 않는다.
   - front matter에서 같은 키를 두 번 선언하지 않는다. 예: `categories` 중복 선언

2. **문서 구조**
   - 자동 목차가 생성되므로 본문 초반에 수동 목차를 추가하지 않는다.
   - 글 제목과 같은 `# 제목`을 본문에 다시 작성하지 않는다.

3. **본문 내용**
   - 도입 요약과 본문에서 같은 의미의 설명을 반복하지 않는다.
   - 글자 수를 늘리기 위한 문장, 예제, 목록을 추가하지 않는다.
   - 같은 단어나 강조 표현을 반복해서 사용하지 않는다.

4. **주제**
   - 기존 `_posts`의 제목, categories, tags를 확인해 이미 다룬 주제인지 점검한다.
   - 같은 주제로 새 글을 작성하려면 다음 중 하나 이상의 차이가 명확해야 한다.
     - 더 최신 버전을 기준으로 설명한다.
     - 다른 기술 조합을 다룬다.
     - 더 구체적인 실무 문제를 해결한다.
     - 이전 글보다 좁고 깊은 예제를 제공한다.

## 품질 기준

1. **길이**
   - 새 포스트의 본문은 샘플 코드를 제외하고 최소 6,000자 이상을 권장한다.
   - 깊이 있는 실전 가이드는 10,000자 이상 작성할 수 있다.
   - 글이 길어질수록 `실전 예제`, `주의사항`, `비교표`, `체크리스트`, `FAQ`처럼 독자에게 실제로 도움이 되는 내용으로 확장한다.

2. **정확성**
   - 버전이 중요한 기술은 버전을 명시한다. 예: `Spring Boot 3`, `Kotlin 2.x`
   - 공식 문서와 다르게 동작할 수 있는 내용은 단정하지 않는다.
   - 성능, 보안, 운영 관련 조언은 적용 조건과 한계를 함께 쓴다.

3. **실무성**
   - 단순 개념 설명에서 끝내지 않고 실제 프로젝트에서 어떻게 쓰는지 보여준다.
   - 설정, 코드, 테스트, 운영, 장애 대응 중 주제에 맞는 관점을 최소 2개 이상 포함한다.
   - 가능하면 `언제 쓰면 좋은가`, `언제 피해야 하는가`를 함께 설명한다.

4. **가독성**
   - 어려운 문장을 길게 이어 쓰지 않는다.
   - 본문 중간에 의미 없는 홍보성 문구를 넣지 않는다.

- 배경과 문제 상황
- 핵심 개념
- 동작 원리
- 간단한 예제
- 자주 하는 오해
- 참고자료

### 실전 구현형

- 목표와 예제 환경
- 의존성 및 설정
- 핵심 코드
- 테스트 방법
- 운영 시 주의사항
- 확장 포인트
- 참고자료

### 비교/선택형

- 비교 대상 소개
- 선택 기준
- 기능/성능/운영 관점 비교표

- 재발 방지 체크리스트
- 참고자료

## AI 포스트 작성 절차

1. `_posts`의 기존 글 제목, categories, tags를 확인하고 중복 여부를 판단한다.
2. 중복이라면 더 좁은 문제나 더 최신 기준으로 주제를 재정의한다.
3. `ruby scripts/new_post.rb "포스트 제목" english-slug`로 파일명과 front matter를 생성한다.
4. 본문 구조를 먼저 잡고, 각 섹션의 목적을 정한다.
5. 개념 설명 후 실전 예제를 작성한다.
6. 주의사항, 실수 사례, 체크리스트를 추가한다.
7. 공식 문서 중심의 참고자료를 추가한다.
8. 제목이 과장되었거나 본문과 어긋나지 않는지 확인한다.
9. 마크다운 문법, 닫는 코드 fence, 코드블록 언어 표기를 점검한다.
10. Mermaid와 표가 있다면 로컬 브라우저에서 실제 렌더링과 모바일 가로폭을 확인한다.

---

## 작성 전 체크리스트

- [ ] 파일 위치와 파일명이 규칙에 맞다.
- [ ] front matter에 `layout`, `title`, `date`, `categories`, `tags`가 있다.
- [ ] front matter 키가 중복 선언되지 않았다.
- [ ] 제목이 구체적이고 과장 표현이 없다.
- [ ] 도입 요약 2~3줄이 있다.
- [ ] 본문에 개념 설명과 실전 예제가 모두 있다.
- [ ] 코드블록에 언어가 명시되어 있다.
- [ ] 코드블록에 줄 번호를 직접 작성하지 않았고 모든 fence가 닫혀 있다.
- [ ] 주요 섹션은 자동 목차에 포함되는 `##`, `###` 헤딩을 사용한다.
- [ ] 수동 목차를 중복해서 작성하지 않았다.
- [ ] Mermaid를 사용했다면 fenced code block 언어가 `mermaid`로 지정되어 있다.
- [ ] 주의사항 또는 자주 하는 실수 섹션이 있다.
- [ ] 결론 및 도움말 섹션이 있다.
- [ ] 참고자료/레퍼런스가 있다.

## 작성 후 점검

- [ ] 같은 설명이 반복되어 분량만 늘어난 부분이 없다.
- [ ] 코드 예제가 본문 설명과 일치한다.
- [ ] 표와 목록 뒤에 해석이 붙어 있다.
- [ ] 공식 문서 링크가 포함되어 있다.
- [ ] 민감 정보나 실제 운영 비밀값이 없다.
- [ ] 마크다운 렌더링을 깨뜨리는 닫히지 않은 코드블록이 없다.
- [ ] 로컬에서 가능하면 `bundle exec jekyll build`로 빌드를 확인한다.
- [ ] Mermaid 다이어그램이 SVG로 렌더링되고 일반 코드블록에만 줄 번호가 표시된다.
- [ ] 표가 본문 폭을 채우며 데스크톱과 모바일에서 페이지 전체의 가로 넘침을 만들지 않는다.
- [ ] 자동 목차의 항목과 실제 `##`, `###` 섹션 순서가 일치한다.
